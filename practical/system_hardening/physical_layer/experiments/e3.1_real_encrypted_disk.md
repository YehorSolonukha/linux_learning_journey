## Attempting to gain access to real ENCRYPTED disk via live USB Boot (Linux)

### Objectives:
- Demonstrate how Full Disk Encryption (FDE) neutralizes the attack by ensuring that data at rest is unreadable and can't be modified
- Demonstrate the fact that data can still be destroyed/encrypted by an attacker
- Highlight the difference between mitigating attacks that target Confidentiality/Integrity vs Availability

### Results
- Encrypted partition (bitlocker) can't be mounted (proper behavior)
- During my attempts to decrypt the disk, I accessed encryption metadata via ```cryptsetup bitlkDump``` and uncovered that GUID of a keyslot matched my Recovery key. Which implies that anyone with physical access to encrypted disk can decrypt it simply by accessing the recovery key. THIS IS EXAMINED IN EXPERIMENT#4
- Because the disk couldn't be decrypted, new experiment is required: this time trying to decrypt the disk using Windows Live Boot

### Environment
- Target: Windows machine with BitLocker enabled (XTS-AES 128-bit).
- Attacker: Live Linux USB

### Execution Steps:
Target Setup
1. Open File Explorer, right-click the C: drive, and select Turn on BitLocker
2. Follow prompts to save Recovery Key
3. Wait until encryption process is completed
4. Verify via terminal:
 ```
 manage-bde -status
 ```

Attacker Setup
1. ref. experiment2
2. Boot the Target machine from a Live Linux USB.
3. install disblocker, ensure cryptsetup in present:
```
sudo apt-get update
sudo apt install disblocker
cryptsetup
```

Gaining Access 
1. Identify the BitLocker partition using lsblk -f.
2. Attempt to mount the partition (results in error - fs not known)
```
sudo mount /dev/<target_partition> /mnt
```
3. Attempt to decrypt the disk using cryptsetup, --readonly also attempted (results in error - recovery key or password don't match)
```
sudo cryptsetup bitlkOpen /dev/<target_partition> my_windows_drive
```
4. Attempt to decrypt the disk using dislocker - results in error
